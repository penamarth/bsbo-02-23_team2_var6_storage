# Рамки: Информационная система склада.

## Cценарий: Регистрация поступления товаров

## Уровень: Задача, определенная пользователем.

## Основной исполнитель: Кладовщик.

### Заинтересованные лица и их требования:

- **Кладовщик:**
  - Хочет быстро и легко зарегистрировать поступление товаров
  - Хочет указать точное место хранения для каждого товара
  - Хочет видеть актуальный остаток товаров на складе после регистрации поступления
  - Хочет использовать сканер штрих-кодов для ускорения ввода данных

- **Менеджер склада:**
  - Хочет обеспечить корректное распределение товаров по местам хранения
  - Хочет получать информацию о поступающих товарах для планирования
  - Хочет контролировать соответствие поступивших товаров документам
  - Хочет видеть аналитику по поступлениям для прогнозирования

- **Система учета:**
  - Требует точного соответствия данных между системой склада и системой учета
  - Требует полного и своевременного обновления данных о товарных запасах
  - Требует полной информации о партиях и сроках годности для товаров с ограниченным сроком

### Предусловия:

- Кладовщик авторизован в системе
- Система имеет актуальные данные о товарах и местах хранения
- Существует накладная на поступление товаров от поставщика
- Доступны свободные места хранения для поступающих товаров
- Сканеры штрих-кодов и терминалы подключены к системе

### Результаты (Постусловия):

- Товары зарегистрированы в системе как поступившие
- Места хранения обновлены с учетом новых поступлений
- Данные о поступлении переданы в систему учета
- Складской остаток обновлен
- История операции зафиксирована для аудита

### Основной успешный сценарий:

1. Кладовщик в системе выбирает опцию "Регистрация поступления товаров"
   - Графический интерфейс отправляет запрос на сервер для получения формы регистрации поступления

2. Сервер возвращает форму регистрации поступления
   - Сервер получает шаблон формы из системы
   - Форма содержит поля для ввода данных поставщика, накладной и товаров

3. Кладовщик вводит данные поставщика и номер накладной
   - В интерфейсе отображается поле для ввода данных
   - Система проверяет наличие поставщика в справочнике

4. Кладовщик добавляет позиции товаров
   - Для каждого товара указывается артикул, количество и партия
   - Система проверяет существование товара в каталоге

5. Система предлагает оптимальное место хранения для каждого товара
   - Система анализирует свободные места хранения
   - Предлагаются подходящие места на основе характеристик товара

6. Кладовщик подтверждает места хранения
   - Для каждого товара указывается конкретное место хранения
   - Система проверяет достаточность свободного места

7. Кладовщик сохраняет поступление
   - Графический интерфейс отправляет данные на сервер
   - Сервер проверяет целостность данных

8. Сервер регистрирует поступление
   - Сервер создает запись о поступлении в системе
   - Система обновляет остатки товаров
   - Система обновляет состояние мест хранения
   - Данные передаются в систему учета

9. Кладовщику показывается подтверждение регистрации
   - Сервер возвращает подтверждение успешной регистрации
   - Отображается информация о зарегистрированном поступлении

### Расширения (альтернативные потоки):

- **Товар не найден в системе**
> 1. Система отображает сообщение о несуществующем товаре  
> 2. Кладовщик может создать новый товар через форму добавления  
> 3. Система запрашивает основные характеристики товара  
> 4. После заполнения данных товар добавляется в каталог  
> 5. Кладовщик продолжает регистрацию поступления с новым товаром

- **Недостаточно места хранения**
> 1. Система отображает предупреждение о недостаточном месте  
> 2. Кладовщик может выбрать другое место хранения вручную  
> 3. Система проверяет выбранное место на соответствие требованиям  
> 4. При подтверждении система резервирует место  
> 5. Если нет подходящих мест, кладовщик может разделить партию и разместить по разным местам

- **Ошибка связи с системой учета**
> 1. Система отображает предупреждение об ошибке связи  
> 2. Кладовщик может сохранить данные локально  
> 3. Система помечает поступление как "ожидающее синхронизации"  
> 4. При восстановлении связи данные автоматически передаются в систему учета

- **Требуется ручное подтверждение**
> 1. Система отображает запрос на подтверждение менеджером  
> 2. Менеджер получает уведомление о необходимости подтверждения  
> 3. После подтверждения менеджером поступление регистрируется окончательно

### Специальные требования:

- Интуитивный интерфейс для сканирования штрих-кодов и автоматического заполнения данных
- Возможность импорта данных из накладной в формате Excel или PDF
- Автоматическое предложение оптимальных мест хранения на основе характеристик товаров
- Проверка соответствия поступивших товаров данным накладной
- Интеграция с системой учета для автоматической синхронизации данных
- Логирование всех действий кладовщика для аудита и контроля
- Поддержка работы с товарами, имеющими партии и сроки годности
- Возможность регистрации частичных поступлений по одной накладной

### Список технологий и типов данных:

- **Данные поступления:** номер накладной, дата, поставщик, список товаров с количеством, партии, места хранения
- **Товары:** артикул, наименование, категория, единица измерения, вес, объем, минимальный и максимальный уровень запаса
- **Партии товаров:** номер партии, дата поступления, срок годности, поставщик
- **Места хранения:** зона, стеллаж, полка, уровень, вместимость, текущая загрузка
- **Система учета:** интеграция через API или файловый обмен
- **Сканеры штрих-кодов:** для автоматического ввода данных о товарах
- **Терминалы:** для работы кладовщиков в зоне склада

### Частота использования:

- Регистрация поступления - высокая частота, ежедневные операции
- Проверка соответствия данных - высокая частота, при каждом поступлении
- Ручное распределение товаров - средняя частота, при отсутствии автоматических предложений
- Работа с партиями товаров - высокая частота для скоропортящихся товаров
